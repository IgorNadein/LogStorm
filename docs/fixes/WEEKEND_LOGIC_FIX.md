# Исправления логики выходных дней

## 🐛 Проблемы, которые были исправлены

### 1. ImportError при запуске GUI
**Проблема:** При запуске `gui_app_fluent.py` возникала ошибка:
```
ImportError: cannot import name 'NIGHT_HOUR_START' from 'config'
```

**Причина:** 
- Была пустая папка `config/` с `__init__.py`, которая перехватывала импорт вместо файла `config.py`
- В `validators/time_validator.py` импортировались константы, но их убрали из импорта

**Решение:**
1. Переименовали папку `config/` в `config_old/`
2. Восстановили импорт `NIGHT_HOUR_START` и `NIGHT_HOUR_END` в `validators/time_validator.py`
3. Добавили комментарии о централизованной настройке в `config.py`

### 2. Неправильное определение технических сбоев для выходных дней
**Проблема:** Когда сотрудник приходил в свой выходной день (например, на 2-3 часа), система помечала это как технический сбой (красный цвет) из-за:
- "Одно появление" - если было только 2 записи (приход и уход)
- "Критическая недоработка" - если отработал меньше 7 часов

**Решение:** Добавили проверку `is_workday` в `analyzers/technical_analyzer.py`:
- **Ночная активность** - только для рабочих дней
- **Одно появление** - только для рабочих дней
- **Критическая недоработка** - только для рабочих дней

### 3. Белый цвет вместо зеленого для переработок в выходные
**Проблема:** В месячных таблицах, если сотрудник приходил в выходной день, ячейка оставалась белой (без форматирования), хотя это переработка.

**Решение:** Добавили отдельный блок в `reporters/excel_formatter.py` для выходных дней:
```python
# Для выходных дней: только переработка (зеленый)
else:
    if status.get('overtime', False):
        cell_in.fill = self.styles['overtime_cell_fill']
        cell_out.fill = self.styles['overtime_cell_fill']
```

## ✅ Текущая логика работы

### Для рабочих дней:
1. **Красный** - технические сбои (всегда приоритет)
2. **Желтый** - опоздания/недоработки
3. **Зеленый** - переработка (>10 часов)

### Для выходных дней:
1. **Красный** - технические сбои (только критическое отсутствие 100%)
2. **Зеленый** - любая работа = переработка
3. **Белый** - никаких событий (не пришел)

### Технические сбои:
- ✅ **Критическое отсутствие (100%)** - проверяется для всех дней
- ✅ **Массовое отсутствие (>90%)** - только для **рабочих дней**
- ✅ **Ночная активность (23:00-03:00)** - только для **рабочих дней**
- ✅ **Одно появление** - только для **рабочих дней**
- ✅ **Экстремальное опоздание (>3ч)** - автоматически только для рабочих дней
- ✅ **Критическая недоработка (<7ч)** - только для **рабочих дней**

## 🔧 Централизованная конфигурация

Все пороговые значения теперь настраиваются в **одном месте** - файле `config.py`:

```python
# Опоздания
LATE_THRESHOLD_MINUTES = 15  # Опоздание учитывается если больше N минут
CRITICAL_LATE_MINUTES = 180  # Критическое опоздание (3+ часа) = сбой системы

# Недоработки
CRITICAL_UNDERWORK_HOURS = 7  # Критическая недоработка: менее N часов
CRITICAL_UNDERWORK_DIFF = 2  # Критическая недоработка: на N+ часов меньше нормы

# Переработки
OVERTIME_THRESHOLD = 10  # Переработка: более N часов в день

# Ночное время
NIGHT_HOUR_START = 23  # Начало ночного времени (час)
NIGHT_HOUR_END = 3  # Конец ночного времени (час)

# Массовые отсутствия
MASS_ABSENCE_THRESHOLD = 0.9  # >90% отсутствуют
CRITICAL_ABSENCE_THRESHOLD = 1.0  # 100% отсутствуют
```

**Где используются:**
- `validators/time_validator.py` - валидация временных данных
- `analyzers/status_analyzer.py` - анализ статусов
- `analyzers/technical_analyzer.py` - определение технических сбоев
- `validators/absence_validator.py` - проверка массовых отсутствий
